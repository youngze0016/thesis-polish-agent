# /style-check — 风格对标

## 设计哲学

> 好论文的章节结构有固定的节奏感：问题→方法→实验→小结，读者翻目录就知道全文在讲什么。
> 风格对标不是机械模仿，而是学习成熟论文的叙事节奏和结构纪律，找出自己论文的差距。

本 skill 基于对优秀论文的结构分析，提炼出一套工科硕士论文的结构与行文规范，用于检测当前论文与规范的偏差。

## Step 0：加载配置

```bash
# 论文修改小组 — 配置加载
CONFIG=""
for candidate in ".thesis/config.json" "../.thesis/config.json" "../../.thesis/config.json"; do
  if [ -f "$candidate" ]; then
    CONFIG="$(cd "$(dirname "$candidate")" && pwd)/$(basename "$candidate")"
    break
  fi
done

if [ -z "$CONFIG" ]; then
  echo "ERROR: 未找到 .thesis/config.json"
  echo "请先运行 /thesis-init 初始化论文项目"
  exit 1
fi

echo "CONFIG: $CONFIG"
THESIS_DIR=$(python3 -c "import json; print(json.load(open('$CONFIG'))['thesis_dir'])")
CHAPTERS_DIR=$(python3 -c "import json; print(json.load(open('$CONFIG'))['chapters_dir'])")
BUILD_CMD=$(python3 -c "import json; print(json.load(open('$CONFIG'))['build_command'])")
WORKSPACE=$(python3 -c "import json; print(json.load(open('$CONFIG'))['workspace_dir'])")
echo "THESIS_DIR: $THESIS_DIR"
echo "CHAPTERS_DIR: $CHAPTERS_DIR"
echo "WORKSPACE: $WORKSPACE"
```

如果 preamble 输出 ERROR，立即停止并提示用户运行 /thesis-init。

## 内置规范：工科硕士论文结构标准

以下规范提炼自已通过答辩的优秀论文，适用于工科类硕士学位论文。

### 规范 1：全文章节骨架

```
第1章 绪论
  1.1 研究背景与意义
  1.2 国内外研究现状分析
    1.2.1 方向A
    1.2.2 方向B
    1.2.3 方向C
    1.2.4 研究现状总结        ← 必须有总结，点明现有不足和本文切入点
  1.3 论文研究内容
  1.4 论文组织结构

第2章 面向XXX的相关技术与理论    ← 标题应点明"为谁服务"
  2.1 技术领域A基本原理
    2.1.1 子技术1
    2.1.2 子技术2
  2.2 技术领域B基本原理
    2.2.1 子技术1
    2.2.2 子技术2
  2.3 技术领域C基本原理
  2.4 本章小结

第3/4章 方法章（每章一个完整工作）
  X.1 问题的提出              ← 先说问题，不是先说方法
  X.2 本章核心方法名称
    X.2.1 子步骤/子模块1
    X.2.2 子步骤/子模块2
    X.2.3 子步骤/子模块3
    X.2.4 子步骤/子模块4
  X.3 实验验证与分析
    X.3.1 实验设置
    X.3.2 对比实验
    X.3.3 消融实验（可选）
  X.4 本章小结

第5章 结论与展望
  5.1 结论
  5.2 展望
```

### 规范 2：章节标题命名

| 规则 | 好的例子 | 坏的例子 |
|------|---------|---------|
| 纯中文，不混英文缩写 | "数据预处理基本原理" | "数据预处理 (Data Preprocessing) 基本原理" |
| 方法章标题 = 完整方法名 | "基于注意力机制的多尺度特征融合目标检测算法" | "第三章方法设计" |
| 理论章标题点明"为谁服务" | "面向多尺度目标检测的相关技术与理论" | "相关理论技术基础" |
| 子节标题说清做什么 | "基于对偶图的权重传播" | "权重传播" |
| 不用冒号做解释 | "权重塌缩问题分析" | "问题分析：权重学习的退化风险" |

### 规范 3：工作章统一结构模板

每个方法章（第3、4章）必须遵循四段式结构：

**X.1 问题的提出**（1-2页）
- 从上一章的结论或局限出发，引出本章要解决的具体问题
- 用具体现象或数据说明问题的严重性
- 点明本章方法的设计思路（但不展开，只给方向）

**X.2 本章方法**（核心，5-15页）
- 子节数 2-4 个，每个子节对应方法的一个独立步骤
- 每个子节至少 1.5 页（约 30 行 tex），不出现 <15 行的碎片子节
- 标题说清"做什么"而非"是什么"（如"基于多尺度特征融合的目标检测"而非"目标检测"）

**X.3 实验验证与分析**（3-8页）
- 必须有"实验设置"独立子节
- 对比实验和消融实验分开
- 分析遵循"指标→原因→图表佐证→影响"四步链

**X.4 本章小结**（0.5页）
- 总结本章解决了什么问题、方法要点、关键实验结论
- 如有局限，坦诚指出
- 自然过渡到下一章（如果不是最后一章）

### 规范 4：节开头段落

每个 `\xsection` 开头必须有一段引导语（3-5句），交代三件事：
1. 这一节要解决什么问题（承上）
2. 这一节的方法/内容概述（本体）
3. 这一节的子节组织（启下）

**好的例子**（参考范本）：
> "由于实际场景中目标尺度差异极大，如图3-1所示，直接使用单一分辨率的特征图进行检测会导致小目标漏检率大幅上升……综上，有必要设计一种多尺度特征融合方法。本节将从特征提取、尺度对齐、注意力融合和检测头设计四个步骤展开。"

**坏的例子**：
> "本节介绍特征融合方法。"（一句话直接进子节）

### 规范 5：行文纪律

| 项目 | 规范 |
|------|------|
| 段落长度 | 4-7 句，不出现 2 句的超短段 |
| 叙述顺序 | 先问题后方案，不先抛结论后补动机 |
| 公式后解释 | 至少两层：形式层（做了什么运算）+ 直觉层（意味着什么） |
| 实验分析 | 四步链：指标→原因→佐证→影响 |
| 节间过渡 | 上一节末尾或下一节开头有 1-2 句衔接 |
| 小结与引言 | 不重复相同的句子 |
| 术语括注 | 核心算法首次标全称，其他不括注 |
| 图表引用 | 每张图/表在正文中被 \ref 且有实质讨论 |

## Step 1：读取论文结构

读取所有章节 tex 文件，提取：
- 全文 `\xchapter`、`\xsection`、`\xsubsection` 结构树
- 每个节的行数
- 每个节开头段落的句数

---

## Step 2：逐项对标检查

按 5 项规范逐一检查，对每项规范输出：符合 ✅ / 偏差 ⚠️ / 严重偏差 ❌

### 检查 1：全文骨架

- 绪论是否有"研究现状总结"子节？
- 理论章标题是否点明"为谁服务"？
- 方法章是否遵循"问题→方法→实验→小结"四段式？
- 结论章是否有"结论"和"展望"两个独立节？

### 检查 2：章节标题

- 扫描所有 `\xsection`/`\xsubsection` 的中文标题参数
- 检查是否有英文缩写混入
- 检查是否有冒号做解释
- 检查方法章标题是否能从标题看出方法全名

### 检查 3：工作章结构

对每个方法章（c3、c4）：
- 是否有"问题提出"类的首节？
- 方法节子节数是否在 2-4 个？
- 是否有 <15 行的碎片子节？
- 实验是否有独立的"实验设置"子节？
- 对比和消融是否分开？
- 是否有本章小结？

### 检查 4：节开头段落

对每个 `\xsection`：
- 标题后第一段有几句话？
- 是否交代了"要解决什么问题"？
- 是否在 1 句话就进了子节？

### 检查 5：行文纪律

- 段落长度分布（统计 <3 句的超短段数量）
- 公式后解释是否充分（`\end{equation}` 后第一段句数）
- 节间是否有过渡

## 安全规则

- **只读模式**：本 skill 不修改任何 tex 文件，只输出检查报告
- 修改建议通过推荐其他 skill 完成

---

## 收尾协议（完成后必须执行）

每次 skill 执行完毕后，无论结果如何，必须执行以下三步：

**1. 更新问题清单 `$WORKSPACE/issues.md`**
- 本次发现的新问题：追加到"待处理"区
- 未变化的问题：不动

**2. 更新进度文件 `$WORKSPACE/progress.md`**
- 在"完成记录"区追加一行

**3. 保存报告（如有输出）**
- 报告保存到 `$WORKSPACE/reports/` 目录
- 文件名格式：`YYYYMMDD-style-check.md`

---

## Completion Status

- **DONE** — 全部 5 项检查完成，报告已输出
- **DONE_WITH_CONCERNS** — 发现严重结构偏差，强烈建议修改
- **BLOCKED** — 章节文件不存在或无法读取
- **NEEDS_CONTEXT** — 需要用户确认某些结构选择是有意为之还是需要修改
